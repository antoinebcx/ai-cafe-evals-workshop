Eval design — a working session
01 · Why evals

Checking a few outputs by hand is how silent regressions ship.

  • There is rarely a single correct answer to assert against. Every input has many valid replies and many wrong ones.
  • Behaviour shifts with every change — a prompt edit, a model upgrade, a different retrieval context. The diff often shows nothing.
  • Manual review does not scale and does not survive a prompt change — but you will still need it to bootstrap and calibrate the eval suite itself.

You need quantitative, multi-case, repeatable measurement.

02 · The reframe

Evals are not a test layer. They are the spec.

  • In deterministic software, code encodes the rules and tests verify them.
  • With LLMs, the model is opaque and probabilistic. The eval suite is the only place where what we want is written down unambiguously.
  • The eval is what forces the team to commit to a single interpretation.
If you do not know what to grade, you do not know what you are building.
03 · The metric menu

Three families. Mix them.

Deterministic
equality · regex · schema · range

Cheap, fast, reproducible. Right for structure, classification, expected fields.

Brittle on phrasing: "30 days" ≠ "30-day" will fail a perfectly good reply.

LLM-as-judge
rubric + small integer scale

Right for subjective dimensions — tone, factuality, intent.

The judge is its own model. Calibrate it against ~20 hand-graded cases before you trust its scores.

Operational
latency · cost · length · tokens

A quality gain that doubles cost and latency is not necessarily a win.

Cost compounds. A run that is cheap in dev can break the budget at production scale.

Most dimensions take a mix: deterministic where there is one right answer, judges where there is taste, operational always. For agents: grade the trajectory — which tools were called, in which order — not only the final answer.

04 · The dataset

Fifty curated cases beat five thousand noisy ones.

An evaluation dataset is a set of inputs, each paired with what good output looks like — sometimes an exact answer, sometimes a rubric.

  • Eval quality is upper-bounded by dataset quality. Hand-pick the seed.
  • Tag every case (happy_path, edge_case, adversarial…) so you read pass rate per slice, not one global number.
  • Every production failure becomes a permanent test case. The dataset compounds.
  • Watch for leakage: do not grade on cases the system has already seen — few-shot examples in your prompt, fine-tuning data, or anything in the model's training set.
05 · What evals unlock

Once the eval is solid, the agent can do the searching.

06 · The loop

An eval suite is never finished.
The dataset grows from production.

Now · 30 minutes

Edit eval.py. Find the failures in forty cases.

  • The system: a naive customer-support agent. Outputs are cached so the eval runs offline — failures are seeded into the data on purpose.
  • The starter: one structural metric that passes 100%. That means nothing is graded yet — that is where you come in.
  • The work: read cases beside outputs, pick three to five dimensions, design a metric for each, run, read failures, iterate.

The design choices are the lesson, not the code. Some failures will be real bugs. Some will be your eval being too strict. Both are signals.
